# jobbly-data-architecture.md
### Jobbly ‚Äî Data Architecture

---

## Principles

- Every commission field is calculated at lead creation and stored immutably ‚Äî never recomputed dynamically
- Contractor rate lives on the lead record, not the campaign ‚Äî it varies per job/package
- Campaign settings (markup %, commission %) apply to future leads only when changed mid-campaign
- Every lead status change is written to the audit log ‚Äî no exceptions
- All non-admin database queries are scoped by campaign ID at the API level
- Quote numbers are generated by Jobbly on lead creation ‚Äî never supplied by external systems

---

## Tables

### 1. `campaigns`

One row per client campaign. The top-level entity everything else belongs to.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `name` | String | Campaign name (e.g. "Continuous Group Guttering") |
| `industry` | String | Industry type (e.g. "guttering", "plumbing") |
| `client_company_name` | String | Client's company name (e.g. "Continuous Group") |
| `subcontractor_company_name` | String | Subcontractor's company name (e.g. "Pro Water Blasting") |
| `markup_percentage` | Decimal | Client markup on contractor rate (e.g. 25.00) |
| `commission_percentage` | Decimal | Omniside's cut of gross markup (e.g. 40.00) |
| `client_margin_percentage` | Decimal | Client's cut of gross markup (e.g. 60.00) ‚Äî must equal 100 minus commission_percentage |
| `status` | Enum | `ACTIVE`, `PAUSED`, `COMPLETED` |
| `start_date` | Date | Campaign start date |
| `created_at` | DateTime | Auto-set on creation |
| `updated_at` | DateTime | Auto-updated on any change |

**Notes:**
- `markup_percentage` and `commission_percentage` are editable by admin at any time
- Changes apply to future leads only ‚Äî existing lead records are unaffected
- `status` is editable both in Campaign Settings and via a quick toggle on the campaign page

---

### 2. `users`

One row per user account. All roles live in this table.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `name` | String | Full name |
| `email` | String | Unique, used for login |
| `password_hash` | String | Hashed password (bcrypt) |
| `role` | Enum | `ADMIN`, `CLIENT`, `SUBCONTRACTOR` |
| `campaign_id` | UUID | FK ‚Üí `campaigns.id`. Null for ADMIN, required for CLIENT and SUBCONTRACTOR |
| `is_active` | Boolean | False = deactivated, cannot log in |
| `created_at` | DateTime | Auto-set on creation |
| `updated_at` | DateTime | Auto-updated on any change |

**Notes:**
- ADMIN users have `campaign_id` as null ‚Äî they access all campaigns via the selector
- CLIENT and SUBCONTRACTOR users are locked to their assigned campaign
- Deactivated users (`is_active: false`) cannot log in but their records are retained for audit purposes

---

### 3. `leads`

One row per lead. The core entity in Jobbly.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `campaign_id` | UUID | FK ‚Üí `campaigns.id` |
| `quote_number` | String | Unique, auto-generated by Jobbly (e.g. "JBL-00001") |
| `customer_name` | String | |
| `customer_phone` | String | |
| `customer_email` | String | Optional |
| `property_address` | String | Full address string |
| `google_maps_url` | String | Auto-generated from `property_address` on lead creation |
| `property_perimeter_m` | Decimal | Metres, from webhook payload |
| `property_area_m2` | Decimal | Square metres, from webhook payload |
| `property_storeys` | Integer | Number of storeys, from webhook payload |
| `contractor_rate` | Decimal | Subcontractor's price for this specific job |
| `customer_price` | Decimal | Stored at creation: `contractor_rate √ó (1 + markup_percentage/100)` |
| `gross_markup` | Decimal | Stored at creation: `customer_price ‚àí contractor_rate` |
| `omniside_commission` | Decimal | Stored at creation: `gross_markup √ó (commission_percentage/100)` |
| `client_margin` | Decimal | Stored at creation: `gross_markup ‚àí omniside_commission` |
| `status` | Enum | `LEAD_RECEIVED`, `QUOTE_SENT`, `JOB_BOOKED`, `JOB_COMPLETED` |
| `source` | String | How the lead entered Jobbly (e.g. "n8n_webhook") |
| `webhook_raw` | JSON | Raw webhook payload stored for debugging and field mapping flexibility |
| `invoice_url` | String | Internal storage path of uploaded invoice file ‚Äî populated automatically after file upload, never entered manually by the user |
| `invoice_uploaded_at` | DateTime | Timestamp of invoice attachment |
| `invoice_uploaded_by` | UUID | FK ‚Üí `users.id` |
| `commission_reconciled` | Boolean | Oli marks true once commission is verified and received |
| `commission_reconciled_at` | DateTime | Timestamp of reconciliation |
| `notes` | Text | Optional internal notes field |
| `created_at` | DateTime | Auto-set on lead creation |
| `updated_at` | DateTime | Auto-updated on any change |

**Notes:**
- `quote_number` format: `JBL-` prefix + zero-padded sequential number per campaign (e.g. JBL-00001, JBL-00002)
- All five financial fields (`contractor_rate`, `customer_price`, `gross_markup`, `omniside_commission`, `client_margin`) are calculated and stored at creation using the campaign's rates at that moment ‚Äî they never change after creation
- `webhook_raw` stores the full raw payload from n8n so field mapping can be adjusted without data loss
- `google_maps_url` is generated as: `https://www.google.com/maps/search/?api=1&query=<url-encoded address>`
- **Invoice upload UX**: The subcontractor never pastes a URL. They click an "Attach Invoice" button which opens a standard file picker on their device (works on mobile and desktop). The file uploads automatically and `invoice_url` is populated behind the scenes. Accepted formats: PDF, JPG, PNG.
- `invoice_url` is required and must be non-null before a status change to `JOB_COMPLETED` is accepted

### 4. `audit_log`

One row per status change. Append-only ‚Äî rows are never updated or deleted.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `lead_id` | UUID | FK ‚Üí `leads.id` |
| `campaign_id` | UUID | FK ‚Üí `campaigns.id` (denormalised for fast campaign-level queries) |
| `changed_by_user_id` | UUID | FK ‚Üí `users.id` |
| `changed_by_name` | String | Stored at write time ‚Äî retained even if user is later deleted |
| `old_status` | Enum | Previous status value |
| `new_status` | Enum | New status value |
| `created_at` | DateTime | Timestamp of the change ‚Äî auto-set, never editable |

**Notes:**
- `changed_by_name` is denormalised (copied from the user record at write time) so the audit trail remains accurate even if a user is renamed or deleted
- This table is append-only ‚Äî no updates, no deletes, ever
- Every status change on a lead must write a row here before the lead record is updated

---

### 5. `attachments`

One row per uploaded file. Currently used for invoices ‚Äî extensible for other file types later.

| Field | Type | Notes |
|---|---|---|
| `id` | UUID | Primary key |
| `lead_id` | UUID | FK ‚Üí `leads.id` |
| `campaign_id` | UUID | FK ‚Üí `campaigns.id` |
| `uploaded_by_user_id` | UUID | FK ‚Üí `users.id` |
| `file_name` | String | Original filename |
| `file_type` | String | MIME type (e.g. "application/pdf", "image/jpeg") |
| `file_url` | String | Storage path or URL |
| `file_size_bytes` | Integer | |
| `attachment_type` | Enum | `INVOICE` (extensible for future types) |
| `created_at` | DateTime | Auto-set on upload |

## Business Logic Rules (Enforced at API Level)

1. A lead's status can only move forward through the pipeline ‚Äî never backwards
2. A lead cannot be marked `JOB_COMPLETED` unless `invoice_url` is non-null
3. Every status change must write to `audit_log` before the lead record updates ‚Äî if the audit log write fails, the status change is rolled back
4. Financial fields on a lead are set once at creation and are never writable again after that point
5. All queries for CLIENT and SUBCONTRACTOR users must include a `WHERE campaign_id = [user.campaign_id]` filter ‚Äî this is enforced in the API route layer, not the UI
6. Quote numbers are generated sequentially per campaign and are never reused, even if a lead is deleted
7. Campaign settings changes (`markup_percentage`, `commission_percentage`) do not trigger recalculation on any existing lead records

## Webhook Field Mapping

The n8n webhook payload field names are not yet finalised. To handle this gracefully:

- The full raw payload is stored in `leads.webhook_raw` (JSON) on every lead
- A field mapping layer in the API route translates incoming webhook fields to Jobbly's internal field names
- Field mappings are documented in `jobbly-webhook-spec.md`
- If a webhook field is missing or unmapped, the lead is still created with nulls for those fields and flagged for review

## Phase 2: Invoice Data Extraction

When a subcontractor uploads an invoice, the ideal flow is:
1. Jobbly parses the invoice (PDF or image) using an AI document parser
2. Extracts the quote number from the invoice content
3. Matches it to the corresponding lead record via `quote_number`
4. Auto-populates `contractor_rate` and `customer_price` on that lead from the invoice figures

For MVP, the VA manually enters the contractor rate when uploading the invoice. The data model already supports auto-population ‚Äî `contractor_rate` on the lead record is the target field. The invoice parsing layer is a Phase 2 addition that slots in without any schema changes.
