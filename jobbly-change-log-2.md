## Change 19 — [#120] "Today" Button on Job Booked Date Picker

### Problem
When a user moves a lead to "Job Booked" status, they must manually select the day, month, and year from three dropdowns. In most cases, the job is being booked for today. Selecting today's date across three dropdowns is unnecessarily slow.

### What needs to happen
Add a "Today" button directly alongside the three date dropdowns in the Job Booked status update modal. Clicking it instantly populates all three dropdowns with today's date — day, month, and year — without the user having to touch the dropdowns manually.

### UI spec
- The "Today" button sits to the right of or directly below the three date dropdowns
- Styled as a small secondary button — not prominent, not the primary action
- Label: "Today"
- On click: reads the current local date and sets all three dropdown values accordingly
- The "Confirm — Move to Job Booked" button must still be enabled after clicking "Today" — it counts as all three dropdowns being filled
- If the user clicks "Today" and then manually changes one of the dropdowns, that is fine — the manually changed value takes precedence

### Implementation notes
- Use `new Date()` to get the current local date at the time of clicking — not a hardcoded date
- Set day, month, and year state values programmatically the same way a manual dropdown selection would
- No API changes required for this — purely a UI addition to the existing status update modal

## Change 20 — [#121] Dashboard Numbers Rework — Add Campaign Revenue Tab and Fix Client View

### Problem
The current dashboard stat cards do not clearly show the campaign revenue (the 25% markup on top of the contractor rate), and the client view does not show enough financial context. The naming of some cards is also confusing. Additionally, commission earned and pending should still show even if not reconciled — as a real-time running total.

### What needs to happen

#### New stat card: "Campaign Revenue"
Add a new stat card to the admin dashboard showing the total gross markup generated across all leads — this is the combined 25% that sits between the contractor rate and what the customer pays. This represents the total revenue generated by the campaign above the subcontractor cost.

- **Label**: "Campaign Revenue (ex GST)"
- **Value**: Sum of `gross_markup` across all leads with a non-null `gross_markup`
- **Who sees it**: ADMIN and CLIENT

This gives both Oli and Continuous Group a clear view of what the campaign has generated above cost — separate from what Oli takes as commission.

#### Rename and clarify existing cards
Rename the following cards to be clearer for all users:

| Old name | New name | Who sees it |
|---|---|---|
| Total Revenue | Total Customer Revenue (ex GST) | ADMIN, CLIENT |
| Commission Earned | My Commission Earned (ex GST) | ADMIN only |
| Commission Pending | My Commission Pending (ex GST) | ADMIN only |

"Total Customer Revenue" = sum of `customer_price` across all leads — the total billed to customers.
"Campaign Revenue" = sum of `gross_markup` — the markup layer above contractor cost.
"My Commission Earned" = sum of `omniside_commission` on reconciled leads.
"My Commission Pending" = sum of `omniside_commission` on completed but unreconciled leads.

#### Commission cards — show regardless of reconciliation status
Commission earned and commission pending must always show their running totals — do not hide or zero these out if nothing has been reconciled yet. The cards reflect real-time financial state at all times.

#### Client view stat cards
The CLIENT role dashboard must show:
- Total leads
- Quotes sent
- Jobs booked
- Jobs completed
- Total Customer Revenue (ex GST)
- Campaign Revenue (ex GST) ← new, now visible to client

The client does NOT see:
- My Commission Earned
- My Commission Pending

This gives Continuous Group a full picture of what the campaign has generated financially without exposing Oli's internal commission split.

#### Date range filter interaction
All stat cards added or renamed in this change must fully respect the dashboard date range filter introduced in Change 18. This means:

- When a date range is selected (e.g. "Last month"), every stat card — including the new Campaign Revenue card and all renamed cards — must recalculate based only on leads within that date range
- "Total Customer Revenue" = sum of `customer_price` for leads with `created_at` within the selected range
- "Campaign Revenue" = sum of `gross_markup` for leads with `created_at` within the selected range
- "My Commission Earned" = sum of `omniside_commission` for reconciled leads with `created_at` within the selected range
- "My Commission Pending" = sum of `omniside_commission` for completed, unreconciled leads with `created_at` within the selected range
- "Total Jobs Revenue" (subcontractor card) = sum of `contractor_rate` for completed leads with `created_at` within the selected range

The API endpoints that power the dashboard stat cards must accept and apply `from` and `to` date parameters for every single card — not just the lead count cards. If any card is not filtering by date range, that is a bug and must be fixed in this session.

## Change 22 — [#123] Auto-Apply Dashboard Filters Without a Filter Button

### Problem
On the main dashboard, after changing the status filter dropdown or the date range selector, the user must manually press a "Filter" button to apply the change. This is an unnecessary extra step and slows down the experience.

### What needs to happen
Remove the explicit "Filter" button entirely. Filters must apply automatically and immediately whenever the user changes any filter value.

### Behaviour
- **Status filter dropdown**: as soon as the user selects a different status (e.g. "Job Booked"), the lead table re-fetches and updates instantly — no button press required
- **Date range selector**: as soon as the user selects a different date range (e.g. "Last month"), both the stat cards and the lead table re-fetch and update instantly
- **Custom date range**: for the "Custom range" option which shows a from/to date picker — apply the filter as soon as both a "from" and "to" date have been selected. Do not apply with only one date selected.
- The loading state (spinner or skeleton) must appear immediately when a filter changes and disappear when data has loaded — so the user knows a refresh is happening
- The URL must update with the new filter values as query parameters on every change — same as before, just without requiring a button press

### Implementation notes
- Use `useEffect` or equivalent to watch filter state and trigger a data re-fetch whenever any filter value changes
- Debounce is not needed for dropdown selections — they are discrete choices, not continuous input
- For custom date range, only trigger the fetch when both `from` and `to` are set

#### 5. Auto-complete job when invoice is uploaded
Currently the VA must: (1) upload the invoice, then (2) separately press "Mark as Job Completed". This is two steps where one will do.

Change the behaviour so that when an invoice is uploaded on a lead that is currently in `JOB_BOOKED` status, the API automatically advances the lead to `JOB_COMPLETED` immediately after the invoice is successfully saved and the financial fields are calculated — but only when the requesting user's role is `SUBCONTRACTOR`.

```
if (invoice uploaded successfully AND lead.status === 'JOB_BOOKED' AND session.user.role === 'SUBCONTRACTOR') {
  → auto-advance status to JOB_COMPLETED
  → write status change to audit_log with the uploading user as the actor

}
```

If the user is ADMIN, the invoice uploads and financials calculate, but the status does not auto-advance — the admin manually controls status as normal.

If the lead is at any status other than `JOB_BOOKED` when the invoice is uploaded (e.g. a replacement invoice on an already-completed job), do not change the status — just update the invoice fields.

- Show a success message to the SUBCONTRACTOR user: "Invoice uploaded and job marked as completed."
- Show a normal success message to the ADMIN user: "Invoice uploaded successfully."

## Change 25 — [#127] Fix Mark Reconciled — Nothing Happens on Confirm

### Problem
On the commission page, when the admin selects a month, clicks "Mark Reconciled", and confirms in the modal — nothing happens. The batch is not created, the leads are not stamped, and the UI does not update. This is a critical bug that makes the entire reconciliation feature non-functional.

### Root cause investigation
Before writing any fix, investigate the following in order:

1. Is the "Confirm Reconciliation" button in the modal correctly calling `POST /api/commission/reconcile`?
2. Is the request body being sent correctly — does it include `month_keys` and `label`?
3. Is the API route `/api/commission/reconcile` receiving the request? Add a console log at the top of the route handler to confirm it is being hit.
4. Is the campaign ID being correctly passed from the session to the API route? A missing or incorrect campaign ID would cause the query to return no leads.
5. Is the database transaction completing successfully? Check for any Prisma errors being silently swallowed.
6. Is the frontend re-fetching the month data after the API call completes? If the API call succeeds but the UI doesn't update, the issue is in the frontend refresh logic.

### What the correct flow must be
1. User selects one or more months on the "By Month" tab
2. User clicks "Mark Reconciled" in the actions bar
3. Confirmation modal opens
4. User clicks "Confirm Reconciliation"
5. `POST /api/commission/reconcile` is called with `{ month_keys: ["2025-03"], label: "March 2025" }`
6. API creates a `ReconciliationBatch` record
7. API stamps `reconciliation_batch_id` on all `JOB_COMPLETED` leads in those months
8. API returns the created batch
9. Frontend re-fetches the months list — the reconciled month disappears from "By Month" tab
10. Frontend re-fetches the batches list — the new batch appears in "Reconciled Batches" tab
11. Stat cards re-calculate and update

Fix whatever is broken in this chain. Do not mark this item complete until the full flow works end to end and can be demonstrated by reconciling a real month.

## Change 27 — [#129] Forgot Password Flow on Login Page

### Problem
Currently there is no way for a user to reset their own password. If they forget it, they must contact Oli, who manually resets it. This is a burden on Oli and a poor user experience. A self-service password reset flow is needed.

### What needs to happen
Add a "Forgot your password?" link on the login page that takes the user through a self-contained password reset flow — no email required, verification is done by confirming the email address exists in the system.

### Flow

**Step 1 — Forgot password link**
On the `/login` page, add a small "Forgot your password?" link below the password input field. Subtle styling — not competing with the Sign In button.

**Step 2 — Reset password page (`/reset-password`)**
Clicking the link navigates to `/reset-password`. This page has:
- Page title: "Reset your password"
- Subtitle: "Enter your email address and we'll let you create a new password."
- Email input field
- "Continue" button

On submit:
- Check if the email exists in the database AND the user is active (`is_active = true`)
- If the email is NOT found or the account is inactive: show a generic message — "If that email is registered, you can set a new password below." — do not confirm whether the email exists (security best practice)
- If the email IS found: show the new password fields (Step 3) on the same page — do not navigate away

**Step 3 — New password fields (shown after email is verified)**
Below the email field (which becomes read-only), show:
- New password input (with eye toggle — same as Change 10)
- Confirm new password input (with eye toggle)
- "Update Password" primary button

Validation rules:
- New password must be at least 6 characters
- New password and confirm new password must match
- New password cannot be the same as the current password (check against bcrypt hash)

On successful update:
- Hash the new password and save it to the user record
- Show success message: "Password updated. You can now log in with your new password."
- Show a "Back to login" link — do not auto-redirect, let the user click

**No email sending required** — the reset flow is entirely on-screen. The user proves they know their registered email address, then sets a new password immediately.

### API endpoints needed

`POST /api/auth/verify-email`
- Body: `{ email }`
- Checks if email exists in users table and `is_active = true`
- Returns `{ exists: true }` or `{ exists: false }` — but the frontend always shows the same message regardless (do not expose whether the email exists in the UI response text)

`PATCH /api/auth/reset-password`
- Body: `{ email, newPassword }`
- Verifies email exists and account is active
- Hashes new password with bcrypt
- Updates user record
- Returns `200` on success, `400` on failure
- Does NOT require an existing session — this is an unauthenticated route

### Security notes
- Rate limit both endpoints: max 5 requests per IP per 15 minutes — prevent brute force
- Never confirm in the UI whether an email address is registered or not
- The same generic message shows whether the email exists or not
- After a successful reset, any existing sessions for that user should be invalidated (force re-login)
